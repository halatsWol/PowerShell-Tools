TOPIC
    about_RepairSystem

SHORT DESCRIPTION
    The `RepairSystem` module provides a function to repair Windows systems locally or remotely by running System File Checker (SFC) and Deployment Imaging Service and Management Tool (DISM) commands, with optional Component Store cleanup, Windows Update Cache cleanup, SCCM cache cleanup, and CCM repair.


LONG DESCRIPTION
--------------
    Repair-System:
    This function performs the following tasks on a specified computer (either locally or a Remote-Device):

    - Pings the remote machine to check its availability.
    - Executes `sfc /scannow` to check and repair system files.
    - Optionally, executes `DISM /Online /Cleanup-Image /ScanHealth` and `DISM /Online /Cleanup-Image /RestoreHealth` to repair the Windows image.
    - Optionally, performs Component Store cleanup and Windows Update Cache cleanup.
    - Optionally, deletes the contents of the CCMCache folder and SoftwareDistribution\Download folder, and/or runs CCMRepair.exe.
    - Prints progress and status to the local console. Step outputs are written to temporary log files, then consolidated into a single master repair log (SystemRepair_<PC>_<date>.log) in CMTrace-compatible format; individual step log files are removed after embedding.
    - On remote runs, transfers the master log and a CBS/DISM system log archive to the local machine.


    Parameters:
    -----------
    - ComputerName (Optional): The hostname or IP address of the remote computer. If not provided, the command will be run locally; if provided, it will run on the specified remote-device.
    - remoteShareDrive (Optional): The ShareDrive of the Remote-Device on which Windows is installed. If none is provided, Default-Value 'C$' will be used.
        The Command `Repair-System -ComputerName SomeDevice -remoteShareDrive D$` will result in Network-Path \\SomeDevice\D$\
    - noSfc (Optional): When specified, the `sfc /scannow` command is skipped.
    - noDism (Optional): When specified, all DISM commands are skipped. This cannot be used in combination with `-IncludeComponentCleanup`.
    - Quiet (Optional): Suppresses console output on the local machine. The output is always logged to files on the remote machine.
    - IncludeComponentCleanup (Optional): Analyzes and, if recommended, cleans up the Windows Component Store. This cannot be used in combination with `-noDism`.
    - WindowsUpdateCleanup (Optional): Renames the `SoftwareDistribution` and `catroot2` folders after stopping related services, effectively performing a Windows Update Cache cleanup. Additionally runs Windows Update and BITS Diagnostics.
    - ChangeTimeout (Optional): Decimal multiplier to change when DISM/SFC and Windows Update Diagnostics will timeout (value `-ChangeTimeout 2` will double the time, `-ChangeTimeout 0.5` will half it). Range = 0.25 - 10.0
    - sccmCleanup (Optional): Deletes the contents of the CCMCache folder and SoftwareDistribution\Download folder.
    - KeepLogs (Optional): Individual step log files are retained alongside the master log instead of being deleted after their content is embedded. On remote runs, step logs remain on the remote device and the full set (step logs + master log) is transferred to the Client.
    - init (Optional): When specified, the Config-File will be Written to the Module-Root-Directory. This will NOT overwrite an existing Config-File. No other parameter will be executed when this is specified.
    - Credentials (Optional): When specified, this will prompt for User-Credentials that will be used for the Remote-Access.
        This can be used e.g. by passing on a credential Object `-Credentials (Get-Credential)`
    - RepairCCM (Optional): When specified, the CCMRepair.exe will be executed. This will also copy the ccmsetup.log to the local Temp-Path.
    - AnalyzeExitCode (Optional): Decodes a previously produced exit code (see EXIT CODES below) into a human-readable, per-step breakdown. Cannot be combined with any other parameter and never performs any repair actions.


EXAMPLES
    Example 1: Repair corrupted system files remotely
    -----------------------------------------------
    PS C:\> Repair-System -ComputerName <remote-device>

    This example repairs corrupted system files using the System File Checker (SFC) tool and runs
    DISM Scan and, if required, DISM Restore on a Remote Machine.


    Example 2: Repair corrupted system files locally
    -----------------------------------------------
    PS C:\> Repair-System

    This example repairs corrupted system files using the System File Checker (SFC) tool and runs
    DISM Scan and, if required, DISM Restore on the Local Machine.


    Example 3: Connect to a non-default Share-Drive
    -----------------------------------------------
    PS C:\> Repair-System -ComputerName SomeDevice -remoteShareDrive D$

    Connects to \\SomeDevice\D$\. This can be used if the SystemRoot (installation of Windows) is either
    not on Drive C:, or if the Share-Drive has a different name.


    Example 4: Run only SFC on the remote computer
    -----------------------------------------------
    PS C:\> Repair-System -ComputerName <remote-device> -noDism

    This example performs the same tasks as Example 1, but skips all DISM commands.


    Example 5: Run SFC and DISM on the remote computer with no console output
    -----------------------------------------------
    PS C:\> Repair-System -ComputerName <remote-device> -Quiet

    Outputs are logged to files but not shown on the console.


    Example 6: Repair corrupted system files remotely with Component Cleanup
    -----------------------------------------------
    PS C:\> Repair-System -ComputerName <remote-device> -IncludeComponentCleanup

    This example performs the same tasks as Example 1 but additionally analyzes the Component Store
    and removes old data which is not required anymore. Cannot be used with '-noDism'.


    Example 7: Perform Windows Update Cache cleanup
    -----------------------------------------------
    PS C:\> Repair-System -ComputerName <remote-device> -WindowsUpdateCleanup

    Stops the Windows Update and related services, renames the SoftwareDistribution and catroot2 folders,
    restarts the services, and runs the Windows Update and BITS Diagnostics.


    Example 8: Decode a previously produced exit code
    -----------------------------------------------
    PS C:\> Repair-System -AnalyzeExitCode "0000000000"

    Decodes the given exit code ("0000000000" = every step succeeded/was not requested) and prints a
    description of each step's result. Runs standalone; performs no repair actions.


    Example 9: Access result properties after a run
    -----------------------------------------------
    PS C:\> Repair-System -noSfc
    PS C:\> $RepairSystemResult.Actions
    PS C:\> $RepairSystemResult.Analysis | Format-Table

    After any run, the full result object is available in $RepairSystemResult. Actions shows which
    steps were requested; Analysis gives a per-step status breakdown.


    Example 10: Inline property access
    -----------------------------------------------
    PS C:\> (Repair-System -ComputerName SomeDevice).DetailedExitCode

    Retrieves only the detailed exit code string from a remote run.


    Example 11: Filter analysis to only active steps
    -----------------------------------------------
    PS C:\> $r = Repair-System
    PS C:\> $r.Analysis | Where-Object { $_.Status -ne 'Not requested' } | Format-Table Position, Label, Status


NOTES
    The script is provided "as-is" and is not supported by Microsoft.
    Test thoroughly before using on production systems, as improper use may cause data loss or system instability.
    Ensure you understand the consequences before making system changes or deleting files.

    !! If in Windows Terminal Settings, Terminal is set as Default Terminal Application, Powershell 5.1 may throw an error instead of prompting for User Credentials.

    Example:
    -----------------------------------------------
    PS C:\> Repair-System -Computername myRemoteDevice1 -Credentials (Get-CredentialObject)


    Requirements:
    -------------
    - WinRM: Must be enabled and configured on the remote computer, to be able to work with `-ComputerName`.
    - Administrative Privileges: Running this script requires administrative privileges on the intended computer.
        For running this for a remote device, Powershell must be started as the user which has administrative privileges on the remote computer.

OUTPUT
    Repair-System returns a RepairSystem.Result object that is suppressed from the default display.
    Access it via assignment, inline property access, or the $RepairSystemResult module global:

        $r = Repair-System ...
        $r.ExitCode             # 0 / 1 / 2
        $r.DetailedExitCode     # e.g. "0000000000"
        $r.ComputerName         # target device
        $r.LogPath              # full path to master log ($null for pre-log early exits)
        $r.Actions              # PSCustomObject — which steps were requested (bool per step):
                                #   SFC, DISMScanHealth, DISMRestoreHealth,
                                #   DISMAnalyzeComponentStore, DISMComponentCleanup,
                                #   SCCMCleanup, WindowsUpdateCleanup, RepairCCM
        $r.Analysis             # array of per-step objects:
                                #   Position, Label, Value, Status
                                #   Status: Success | Not requested | Skipped (connection lost) |
                                #           Success (restart required) | Timed out |
                                #           Terminated externally | <known-code description>

    The last result is also stored in $global:RepairSystemResult after every run.

    -AnalyzeExitCode does not return an object; it writes directly to the host.


EXIT CODES
    $LASTEXITCODE after Repair-System runs is a conventional single digit, suitable for scripts/CI/batch to branch on:

        0 = Full success - every step succeeded or was not requested.
        1 = The run completed (possibly only partially, e.g. a mid-run connection loss) but one or
            more steps reported a problem.
        2 = A startup/fatal error meant no repair steps ran at all (bad parameters, target
            unreachable, WinRM failure, not elevated, config error, conflicting parameters).

    The full, lossless detail behind that digit is printed to the console as "Detailed Exit Code:
    <code>" and returned as the DetailedExitCode property of the result object (see OUTPUT above).

    The detailed code is made up of one field per step, concatenated in a fixed position order (no
    reordering, no delimiters). Each field starts with a single hex digit (0-8) giving how many hex
    digits follow ('0' alone means the step's value is 0), then those digits (if any) are the step's
    real return value (DISM/SFC's own exit code, or the step's own small result code). A fully
    successful run collapses to "0000000000" (ten '0' characters).

    Step positions:
        0 = Startup / Pre-Flight Checks (parameter/network/WinRM/elevation/config errors, or
            connection-lost mid-run)
        1 = SFC /scannow
        2 = DISM ScanHealth
        3 = DISM RestoreHealth
        4 = DISM AnalyzeComponentStore
        5 = DISM StartComponentCleanup
        6 = SCCM Cleanup
        7 = Windows Update Cleanup
        8 = Repair CCM
        9 = Zip CBS/DISM Logs

    If a step was not requested, or was skipped because the remote connection was already lost, its
    field is 0. A field value of 5 means the step was requested but skipped because the remote
    connection was lost while/after the previous step ran.

    For the SFC and DISM steps (positions 1-5), a raw process exit code is only trusted if the
    process actually had a fair chance to run: if Repair-System itself killed it for exceeding its
    time budget, that field reads 4294967294 ("timed out"); if it disappeared in well under 30
    seconds without Repair-System killing it, that field reads 4294967293 ("likely terminated
    externally, e.g. via Task Manager; its own exit code could not be trusted").

    Run `Repair-System -AnalyzeExitCode <code>` to get a readable explanation of any code produced
    by a previous run.
